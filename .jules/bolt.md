## 2024-05-16 - [Optimize ScheduleJob update in Dispatcher]
**Learning:** In a highly concurrent background worker environment, fetching a record via ORM (`get_by_pk`) merely to update its status is inefficient and adds unnecessary database roundtrips. SQLAlchemy's direct `update().where().values()` eliminates this bottleneck without sacrificing much readability. However, mocking `update()` with `MagicMock` in unit tests can lead to `ArgumentError: subject table for an INSERT, UPDATE or DELETE expected` because `update()` expects a real SQLAlchemy construct (like a Table or a Declarative Model).
**Action:** Use an in-place `update` statement for single-record status updates where we don't need to interact with the full Python object. When writing tests to assert these types of updates, either use an actual DB session (Integration Test) or mock the final `session.execute` to capture the statement, and patch `update()` carefully by mapping its chain return values to avoid touching real DB objects.
